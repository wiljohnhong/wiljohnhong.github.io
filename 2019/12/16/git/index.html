Git 备忘

Dec 16 2019 8 minutes read (About 1239 words)

Git 是一个开源的分布式版本控制系统。温习一下 Git 操作对于提升工作效率确实很有帮助。

本地基本操作

创建仓库:

将一个文件添加至暂存区

将一个文件从暂存区中移除（同时也会从磁盘上移除）

提交改动

检查仓库状态

展示工作区和暂存区的不同之处

查看提交历史

回退到某个提交版本

显示过去执行的命令（撤销改动时很有用）

丢弃工作区（还未被add到暂存区）的改动

丢弃暂存区中的改动

分支管理

查看所有分支

创建并切换分支

合并某分支到当前分支

删除分支

对于未被合并的分支，删除可能造成修改内容的丢失，需要使用大写参数-D

合并分支时可能会使用fast forward模式进行合并，仅仅移动了分支指针，看不出做过的改动。加上--no-ff参数可以用普通模式合并，将合并操作作为一次提交写入历史，所以通常还要用-m对提交进行说明

切换分支时保存现场（即把暂存区和未保存的部分存到当前分支下，否则改动会随分支切换而转移），常见于临时进行其他与当前分支无关的改动（如修bug）

需要恢复现场时先查看可恢复的现场列表

恢复某个现场并删除记录

pop命令相当于连续分别执行恢复apply和删除drop

在某一分支上修复bug后可能要将改动也重现在当前分支上，使用cherry-pick命令可以将一个特定的提交复制到当前分支

分支开多了后，为了merge之后提交历史看起来更清晰，我们希望将提交历史整理成直线，可以使用rebase命令。但是注意，rebase仅可以把本地未push的分叉提交历史整理成直线

远程仓库

为本地仓库关联一个远程仓库（如在GItHub上新创建的仓库）

查看远程仓库的信息，-v显示更详细的信息

进行第一次推送，例如推送本地master分支的所有内容

如果要从已有的远程仓库克隆代码来创建本地仓库

克隆后默认只能看到本地的master分支，如果要在其他分支如dev进行修改则要创建远程的dev分支到本地

推送本地的dev分支到远程仓库也类似地是

多人协作时常常会因为他人更新而推送失败，需要先试图先用pull合并，如果遇到冲突则还要解决冲突，最后再push到远程仓库

而对于非master分支，在第一次pull时会因为没有指定本地dev分支与远程origin/dev分支的链接而失败，需要设置关联如下

标签

为当前分支上最近一次提交（即HEAD）打上标签

对指定的提交打上标签，-a指定标签名，-m指定说明文字

查看所有标签

向远程仓库推送一个标签

推送所有未被推送过的标签

删除本地仓库的标签

将删除推送到远程仓库

配置

全局自报家门

或者直接修改全局配置文件，路径为~/.gitconfig

在配置文件中还可以使用别名，常用的配置是优化git log输出格式的别名

本地仓库的配置文件路径为仓库根目录下的.git/config
